## 执行指令

进入 `knowledge/teaching-guides/`，为每个章节创建教学指引文件。

**核心原则**：一个 `.teaching.md` 文件 = 教材的一个**最小子节**（即没有下级标题的那一层）。

**执行流程**：采用**并行生成 + 上位审查**模式，每批 3 章，最多 3 个 Agent 并行。

```text
┌──────────────────────────────────────────────────────────────┐
│  上位 Agent（调度层）                                         │
│  ─ 将教材章节分批，每批 3 章                                  │
│  ─ 同时启动最多 3 个下位 Agent（并行调用）                    │
│  ─ 每个 Agent 的 prompt：该批教材原文 + TEMPLATE.md + 规范    │
└──────┬───────────────────┬───────────────────┬───────────────┘
       ▼                   ▼                   ▼
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ 下位 Agent 1 │  │ 下位 Agent 2 │  │ 下位 Agent 3 │
│ 批次 1 (1-3) │  │ 批次 2 (4-6) │  │ 批次 3 (7-9) │
│ 生成 .teach  │  │ 生成 .teach  │  │ 生成 .teach  │
└──────┬───────┘  └──────┬───────┘  └──────┬───────┘
       └──────────────────┼──────────────────┘
                          ▼
┌──────────────────────────────────────────────────────────────┐
│  上位 Agent 审查（全部批次返回后统一审查）                     │
│  ─ 逐批逐份对照教材原文，检查：                               │
│    ① 知识点分层是否合理（核心/重要/了解 是否归对层级）       │
│    ② 讲解要求是否准确（是否遗漏关键点或过度展开）            │
│    ③ 出题权限是否正确（正式题/课堂识别/否 是否匹配）         │
│    ④ 原文依据是否精确（段落号是否对应、内容是否忠实）        │
│  ─ 每批给出综合评分（满分 100）                               │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
               批次评分全部 ≥ 95? ──是──→ 通过，阶段 H 结束
                           │否
                           ▼
               ┌───────────────────────────┐
               │ 仅对评分 < 95 的批次：     │
               │ ① 指出具体错误点          │
               │ ② 对应文件和段落          │
               │ ③ 修正方向                │
               │ 重新调用该批次 Agent 修正  │
               └─────────────┬─────────────┘
                             ▼
               下位 Agent 修正 → 重新提交 → 上位再审
               （循环直到所有批次评分 ≥ 95）
```

**分批规则**：
- 每批 3 章（如 chapter-01 ~ chapter-03）
- 末尾不足 3 章时全部发送（如剩 2 章则一批完成）
- 每轮最多并行启动 3 个 Agent
- 超过 3 批时，先并行处理前 3 批，通过后再处理下一轮
- 全部章节通过后，阶段 H 结束

**并行执行要点**：
- 每个 Agent 独立写入各自的 `chapter-XX/` 目录，互不冲突
- 上位审查在所有 Agent 返回后统一进行
- 打回修正时，仅重新调用评分 < 95 的批次对应的 Agent，已通过的批次不重新处理

**审查评分标准**：

| 维度 | 权重 | 评分要点 |
|---|---|---|
| 知识点分层 | 30% | 核心/重要/了解 是否归对层级，有无把"了解即可"升格为核心 |
| 讲解要求 | 25% | 是否覆盖该单元的关键定义、公式、模型；有无遗漏或过度展开 |
| 出题权限 | 20% | 正式题/课堂识别不计分/否 是否与知识点层级匹配 |
| 原文依据 | 25% | 段落号是否精确对应教材，内容是否忠实于原文而非凭空编写 |

**打回修正规则**：
- 上位 Agent 必须指出具体错误点（哪个文件、哪个字段、错在哪里）
- 给出修正方向（如"该知识点应从核心降为重要背景，因为教材仅一句话带过"）
- 下位 Claude 只修正被指出的问题，不改动已通过的部分
- 修正后重新提交，上位再次审查，直到评分 ≥ 95

**决策树 — 是否需要拆文件**：

```text
该标题下面有下级标题（### 或 ####）？
├── 有 → 不拆，等子标题各自独立成文件
└── 没有 → 这就是最小单元，生成一个 .teaching.md
```

**文件命名公式**：
```
{章}-{节}[-{子节}[-{子子节}]]-{英文名}.teaching.md
```

每一级数字对应教材标题的层级：

| 教材标题层级 | 对应编号段 | 说明 |
|---|---|---|
| `# 第X章` | `XX-` | 两位数，从 01 起 |
| `## X.Y 标题` | `XX-YY-` | 两位数，`00` = 章引言，`01` = 第一节 |
| `### X.Y.Z 标题` | `XX-YY-ZZ-` | 两位数，仅当该节有 `###` 子标题时才加 |
| `#### X.Y.Z.W 标题` | `XX-YY-ZZ-WW-` | 同理，仅当有 `####` 时才加 |

**示例 — 教材结构与生成文件的映射**：

教材结构：
```text
# 第3章 线性模型
## 3.1 线性判别函数          ← 有 ### 子标题，不单独成文件
  ### 3.1.1 二分类           ← 无下级标题，最小单元
  ### 3.1.2 多分类           ← 无下级标题，最小单元
## 3.4 感知器                ← 有 ### 子标题，不单独成文件
  ### 3.4.1 参数学习         ← 无下级标题，最小单元
  ### 3.4.4 扩展到多分类     ← 有 #### 子标题，不单独成文件
    #### 3.4.4.1 广义感知器收敛 ← 无下级标题，最小单元
```

**个性化内容填充规则**：

每个 `.teaching.md` 文件按 TEMPLATE.md 模板填写，内容从该单元对应的教材段落中提取：

| 模板字段 | 内容来源 |
|---|---|
| 原文定位 | 从教材中定位该单元的起止段落号 |
| 本节教学目标 | 根据教材内容推断学完后应能做什么 |
| 核心知识点 | 教材中定义、公式、模型、算法等必须掌握的内容 |
| 重要背景 | 帮助理解核心知识点但不做深推导的内容 |
| 了解即可 | 教材中提及但不要求深入的内容（人名、年份、工具名等） |
| 出题权限 | 核心知识点→正式题；重要背景→课堂识别不计分；了解即可→否 |
| 本节不要求 | 明确该单元不涉及的后文内容、外部资料、代码实现等 |
| 覆盖检查模板 | 根据教学目标生成 checklist |

**关键原则**：
- 每个教学指引必须指向教材段落号（`¶XXXX`），不能凭空编写
- 核心知识点的 `出题权限` 只能填 `正式题`、`课堂识别不计分`、`否`
- 不要把后文章节的内容提前纳入当前单元的核心知识点
- 考点名可以是中文（如 `02-04-01-惯性系.teaching.md`），保持与教材术语一致

## 双 Agent 实现

运行本 skill 的 Claude 就是**上位模型**，它通过 Agent 工具同时调用多个 Claude 实例作为**下位模型**。上位模型并行调度、统一审查。

**角色分工**：

| 角色 | 是谁 | 职责 |
|---|---|---|
| 上位模型 | 运行 skill 的 Claude（Claude Code 自身） | 分批调度、并行启动 Agent、组装 prompt、审查输出、评分、决定通过或打回 |
| 下位模型 | Agent 工具调用的 Claude（使用用户配置的 model_id） | 通读教材原文、识别最小单元、生成 .teaching.md 文件、根据反馈修正 |

**并行交流流程**：

```
上位模型                    Agent 1            Agent 2            Agent 3
   │                          │                  │                  │
   │ ① 并行发送：             │                  │                  │
   │   批次1教材+模板  ──────→│                  │                  │
   │   批次2教材+模板  ──────────────────────→  │                  │
   │   批次3教材+模板  ──────────────────────────────────────→  │
   │                          │                  │                  │
   │                    ② 并行生成               │                  │
   │                    .teaching.md        .teaching.md        .teaching.md
   │                    识别最小单元         识别最小单元         识别最小单元
   │                    填充模板内容         填充模板内容         填充模板内容
   │                          │                  │                  │
   │ ③ 等待全部返回           │                  │                  │
   │ ←────────────────────────┼──────────────────┼──────────────────┘
   │                          │                  │
   │ ④ 统一审查：逐批逐份对照教材原文            │
   │    4 维度评分（分层/讲解/出题/原文）         │
   │                                               │
   │  评分 ≥ 95?                                   │
   │  ├─ 全部通过 → 阶段 H 结束                    │
   │  └─ 有批次 < 95 ↓                             │
   │                                               │
   │ ⑤ 仅对未通过批次：                            │
   │    发送具体错误点 + 文件名 + 修正方向         │
   │    重新调用该批次 Agent                        │
   │                                               │
   │ ⑥ 修正后重新审查 → 循环直到全部 ≥ 95         │
   │                                               │
   │  Hook 校验文件结构                            │
   │  通过 → 阶段 H 结束                           │
```

**Agent 工具调用参数**：
- `subagent_type=general-purpose`
- `model=` 从 `COURSE_DIR/pipeline-progress.md` 的"模型配置"段落读取 `model_id`，然后映射为 Agent 工具支持的枚举值（`sonnet`/`opus`/`haiku`）
- 如果用户配置了私有部署模型（Agent 工具无法直接指定），则省略 `model` 参数，使用 Claude Code 当前默认模型

**模型配置说明**：
- Agent 工具继承 Claude Code 当前的 API 配置（api_key、base_url）
- 如果下位模型使用 Anthropic 官方 API：无需额外配置，直接设置 `model=` 即可
- 如果下位模型使用私有部署或其他 API：用户需在启动 skill 前通过 ccswitch 配置目标模型，ccswitch 会切换 Claude Code 的 API 环境且不影响全局配置
- 模型配置在启动协议阶段已收集并写入进度文件，此处直接使用，不再询问用户

**prompt 组装规则**（上位模型发给下位模型）：
- 包含：本批 3 章的教材原文 + TEMPLATE.md 内容 + 文件命名规范 + 已有填充示例
- 不包含：其他批次的内容、评分标准、上位模型的内部逻辑

**反馈规则**（上位模型打回时）：
- 必须指出具体错误（哪个文件、哪个字段、错在哪里）
- 必须给出修正方向（如"该知识点应从核心降为重要背景"）
- 下位模型只修正被指出的问题，不改动已通过的部分

## 质量门

- 每个文件包含完整的 5 个段落（原文定位、教学目标、知识点分层、本节不要求、覆盖检查）
- 核心知识点表格有 4 列且出题权限合法
- 所有知识点有 `¶XXXX` 段落号引用
- 上位审查评分 ≥ 95
- Hook: validate-teaching-guide 通过
