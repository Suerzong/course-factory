# 阶段 B：结构读取

> 摘自 guide/guide.md。本文件是该阶段的完整执行指令。

## 需要的变量

无额外变量。使用阶段 A 收集的变量。

## 前置检查

执行本阶段前，读取 `COURSE_DIR/pipeline-progress.md`：
- 确认阶段 B 状态为"待执行"
- 如果状态为"已完成"，跳过本阶段，直接进入阶段转换
- 如果状态不是"待执行"也不是"已完成"，报错并停止

## 执行指令

读取 `COURSE_DIR/textbook/chapters/总教材.md`，识别教材结构：
- 按一级标题（`#`）切分章
- 统计每章内的 `## / ### / ####` 层级分布
- 输出：章节数量、每章标题、每章起止行号、每章的 `##` 和 `###` 数量

**运行模式分支**：
- `interactive`：
  - 将结构信息展示给用户
  - 必须收到用户明确确认后才能继续
- `batch`：
  - 若结构识别唯一且无阻断性歧义，直接继续
  - 若存在阻断性歧义（如章边界冲突、同级标题混乱到无法稳定切分），报错并停止
  - 非阻断性异常仅记录到 `pipeline-execution.log`

**本阶段只做结构草图**：
- 不生成教学指引
- 不生成学习路径
- 不生成 `unit-manifest.json`
- 只是为后续阶段建立稳定的章节视图

## 质量门

- 识别出的章节数量 >= 1
- 每章有明确的标题和起止行号
- 每章的 `## / ### / ####` 数量可统计
- `interactive` 模式下用户已确认结构无误
- `batch` 模式下不存在阻断性歧义

## 完成记录

质量门通过后，更新 `COURSE_DIR/pipeline-progress.md`：
- 将阶段 B 状态从"待执行"改为"已完成"
- 记录完成时间
- 记录识别出的章节数量和每章标题

## 阶段转换

完成记录写入后：
1. 读取 `COURSE_DIR/pipeline-progress.md`
2. 找到第一个状态为"待执行"的阶段
3. 如果找到 -> 读取对应的 `GUIDE_ROOT/pipeline/stage-{x}.md`，执行该阶段
4. 如果没有"待执行"阶段 -> 所有阶段完成，进入清理协议
